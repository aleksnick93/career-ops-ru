## Система оценки

Оценка использует 6 блоков (A-F) с глобальной оценкой 1-5:

| Измерение | Что оценивает |
|-----------|---------------|
| Match с CV | Навыки, опыт, соответствие proof points |
| Соответствие North Star | Насколько вакансия вписывается в целевые архетипы пользователя (из `_profile.md`) |
| Компенсация | Зарплата vs рынок (5=верхний квартиль, 1=значительно ниже) |
| Культурные сигналы | Культура компании, рост, стабильность, политика удалённой работы |
| Red flags | Блокеры, предупреждения (негативные корректировки) |
| **Global** | Взвешенное среднее вышеперечисленного |

**Интерпретация оценки:**
- 4.5+ → Сильное совпадение, рекомендуется откликнуться немедленно
- 4.0-4.4 → Хорошее совпадение, стоит откликнуться
- 3.5-3.9 → Нормально, но не идеально, откликаться только при конкретной причине
- Ниже 3.5 → Не рекомендуется откликаться (см. Ethical Use в CLAUDE.md)

### Российский рынок -- Специфика (ВАЖНО)

В вакансиях и переговорах на российском рынке есть термины и практики, которых нет в EN/ES/DE/FR/PT. Их НЕОБХОДИМО корректно оценивать:

| Термин | Значение | Влияние на оценку |
|--------|----------|-------------------|
| **КТК** (Трудовой договор по ТК РФ) | Стандартный трудовой контракт | Включает оплачиваемый отпуск (28 календарных дней), больничный, декретные, выходное пособие. Стандарт рынка |
| **ГПХ** (Гражданско-правовой характер) | Договор подряда / оказания услуг | Ставка выше, но без отпуска, больничного, выходного пособия. Считать эквивалент КТК для корректного сравнения |
| **ИП / Самозанятый** | Индивидуальный предприниматель / НПД | Налоговая ставка ниже (6% / 4-6%), но нет социальных гарантий. Уточнить реальную нагрузку |
| **Белая зарплата** | Полностью официальная, задекларированная в трудовом | Стандарт для IT. Если не полностью белая — серьёзный red flag |
| **Серая зарплата** | Часть зарплаты в конверте | Red flag. Проблемы с кредитами, пенсией, судебной защитой |
| **ДМС** (Добровольное медицинское страхование) | Корпоративная медстраховка | Ключевой бенефит в РФ. Без ДМС в крупной компании = red flag. Проверить: со дня выхода или после испытательного? |
| **Испытательный срок** | Стандартно 3 месяца (до 6 для руководителей) | Стандарт рынка, не red flag. Флаговать если > 3 месяцев для линейных специалистов |
| **Отпуск** | 28 календарных дней по закону | Минимум по ТК РФ. Некоторые компании дают 30-35. Меньше 28 = нарушение закона |
| **Gross / Net** | Зарплата до / после НДФЛ (13%) | В объявлениях часто указывают gross. При сравнении пересчитывать: net = gross × 0.87 |
| **Годовой бонус / Премия** | Дополнительная выплата по итогам года | Может составлять 1-3 оклада. Уточнить: гарантированная или по KPI? |
| **Опционы / RSU / ESOP** | Участие в капитале компании | Распространены в международных и крупных tech-компаниях. Проверить vesting, cliff, ликвидность |
| **IT-аккредитация** | Компания включена в реестр аккредитованных IT-организаций Минцифры | Льготы для сотрудников: отсрочка от армии, льготная ипотека, пониженные страховые взносы. Сильный плюс |

### Скрипты переговоров

<!-- [НАСТРОИТЬ] Адаптируйте под вашу ситуацию -->

**Зарплатные ожидания (общий framework):**
> "На основе актуальных данных рынка для этой позиции, мои ожидания в диапазоне [ДИАПАЗОН из profile.yml]. Готов обсуждать структуру — главное для меня полный пакет и перспективы роста."

**Pushback против географического дисконта:**
> "Вакансии, в которых я участвую, ориентированы на результат, а не на локацию. Мой track record не меняется с почтовым индексом."

**Когда предложение ниже цели:**
> "Сейчас я сравниваю с предложениями в диапазоне [более высокая вилка]. [Компания] привлекает меня из-за [причина]. Возможно ли выйти на [целевое значение]?"

**КТК vs ГПХ:**
> "Чтобы корректно сравнить предложения, мне нужно понять полную структуру: оклад gross, ДМС, бонусы, опционы. Если формат ГПХ — какая ставка с учётом того, что отпуск и больничный за свой счёт?"

### Политика локации (Location Policy)

**В формах:**
- Бинарные вопросы "Готовы ли вы работать в офисе?": отвечать в соответствии с реальной доступностью из `profile.yml`
- В текстовых полях: указать часовой пояс и доступность явно

**В оценках (Scoring):**
- Измерение remote при гибриде вне вашего региона/страны: Score **3.0** (не 1.0)
- Score 1.0 только если вакансия явно говорит "обязательно присутствие 4-5 дней/неделю, без исключений"

### Приоритет Time-to-Offer
- Работающее демо + метрики > перфекционизм
- Откликнуться быстрее > изучить глубже
- Подход 80/20, всё с дедлайнами

## Глобальные правила

### НИКОГДА

1. Придумывать опыт или метрики
2. Изменять `cv.md` или файлы портфолио
3. Отправлять отклики от имени кандидата
4. Делиться номером телефона в генерируемых сообщениях
5. Рекомендовать компенсацию ниже рынка
6. Генерировать PDF без предварительного прочтения описания вакансии
7. Использовать корпоративный жаргон
8. Игнорировать tracker (каждая оценённая вакансия регистрируется)

### ВСЕГДА

0. **Сопроводительное письмо:** Если форма позволяет приложить или написать письмо, ВСЕГДА его включать. PDF в том же визуальном дизайне, что и резюме. Содержание: цитаты из описания вакансии, привязанные к proof points, ссылки на релевантные case studies. Максимум 1 страница.
1. Читать `cv.md`, `_profile.md` и `article-digest.md` (если существует) перед оценкой любой вакансии
1b. **При первой оценке каждой сессии:** Выполнить `node cv-sync-check.mjs` через Bash. При предупреждениях — уведомить кандидата перед продолжением
2. Определить архетип вакансии и адаптировать позиционирование согласно `_profile.md`
3. При матчинге цитировать точные строки из резюме
4. Использовать WebSearch для данных о компенсации и компании
5. Регистрировать в tracker после каждой оценки
6. Генерировать контент на языке описания вакансии (русский по умолчанию)
7. Быть прямым и конкретным — без воды
8. Генерируя текст на русском (PDF summaries, bullets, сообщения LinkedIn, истории STAR): естественный русский tech-язык, не дословный перевод. Короткие предложения, глаголы действия, избегать страдательного залога. Технические термины (stack, pipeline, deployment, embedding) не переводить принудительно
8b. **URL case studies в PDF Professional Summary:** Если PDF упоминает case studies или демо, URL ДОЛЖНЫ появиться уже в первом абзаце (Professional Summary). Рекрутеры часто читают только summary. Все URL в HTML с `white-space: nowrap`
9. **Записи в tracker как TSV** — НИКОГДА не редактировать `applications.md` напрямую для новых записей. Писать TSV в `batch/tracker-additions/`, `merge-tracker.mjs` позаботится о слиянии
10. **Включать `**URL:**` в каждый заголовок report** — между Score и PDF
